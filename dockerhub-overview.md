# Boggrt

Boggrt is a containerized mock API server that returns configurable JSON responses based on request method, path, and optional body conditions. 
It is designed for local development, automated integration tests, CI pipelines, and Docker Compose setups where you need a predictable stand-in for external APIs.

You can hand-write JSON endpoint definitions or seed an entire mock API automatically from an OpenAPI 3.x specification.

## Quick reference

- Image: `norcodedev/boggrt`
- JSON endpoints directory: `/resources` (override with `BOGGRT_SOURCE`)
- OpenAPI specifications directory: `/openapi` (override with `BOGGRT_OPENAPI_SOURCE`)
- JSON input format: one or more `.json` files, each containing a single endpoint object or an array of endpoint objects
- OpenAPI input format: `.yaml`, `.yml`, or `.json` OpenAPI 3.x documents
- Source and documentation: [GitHub repository](https://github.com/norcode-dev/boggrt)

## Why use Boggrt?

- Mock HTTP APIs without writing custom stub servers
- Match requests by `method` and `path`
- Add request-body conditions using nested fields, array indexes, and wildcards like `items[*].sku`
- Return custom JSON bodies with custom HTTP status codes
- Seed a complete mock API straight from an OpenAPI 3.x spec, no hand-written endpoints required
- Reuse the same mock files in standalone Docker runs, Testcontainers, and Docker Compose

## How it works

Boggrt loads endpoint definitions from the configured folder and evaluates each incoming request in this order:

1. Match request `method` and `path`
2. If conditions are configured, parse the request body as JSON and evaluate all conditions
3. Return the configured JSON response when everything matches

All configured conditions are combined with logical `AND`.

Behavior summary:

- Default success status is HTTP `200` when `responseStatus` is omitted
- No matching endpoint returns HTTP `404` with `Response not found.`
- A matching path with the wrong HTTP method returns HTTP `405 Method Not Allowed`
- If conditions are configured, a failed condition or invalid JSON body returns HTTP `404`
- Successful responses are returned as `application/json`

## Running the image

```bash
docker run --rm -p 8080:8080 \
  -v "$(pwd)/mocks:/resources" \
  norcodedev/boggrt
```

To load configuration from a different folder inside the container:

```bash
docker run --rm -p 8080:8080 \
  -e BOGGRT_SOURCE=/config \
  -v "$(pwd)/mocks:/config" \
  norcodedev/boggrt
```

To seed endpoints from an OpenAPI specification, mount it into `/openapi`:

```bash
docker run --rm -p 8080:8080 \
  -v "$(pwd)/openapi:/openapi" \
  norcodedev/boggrt
```

You can mount both directories at once to combine generated endpoints with hand-written JSON overrides.

## Example configuration

Simple endpoint:

```json
{
  "method": "GET",
  "path": "/health/mock",
  "response": {
    "status": "ok"
  }
}
```

Conditional endpoint:

```json
{
  "method": "POST",
  "path": "/orders/validate",
  "conditions": [
    { "field": "customer.lastName", "operator": "equals", "value": "Doe" },
    { "field": "items[*].sku", "operator": "contains", "value": "SKU-" }
  ],
  "responseStatus": 202,
  "response": {
    "status": "accepted",
    "code": "MOCK-OK"
  }
}
```

Place these files in the mounted configuration directory and call the matching routes with your HTTP client or test suite.

## OpenAPI specification support

Boggrt can build mock endpoints directly from an OpenAPI 3.x document, so you can stand up a full mock API without writing endpoint files by hand. Drop one or more `.yaml`, `.yml`, or `.json` OpenAPI specs into the `/openapi` directory (override with `BOGGRT_OPENAPI_SOURCE`) and Boggrt registers a route for every `(path, operation)` it finds.

For each operation, Boggrt generates:

- `method` from the operation key (`get`, `post`, ...)
- `path` from the spec path, translating `{name}` parameters to Vert.x style (`/pets/{petId}` â†’ `/pets/:petId`)
- `responseStatus` from the smallest declared `2xx` status, falling back to `200`
- `response` as a JSON body sampled from the response's `application/json` schema

Response bodies are generated by walking the schema: a schema-level `example` or `default` is used verbatim, otherwise the first `enum` value, otherwise a realistic random value with format-aware handling for `email`, `uuid`, `date`, `date-time`, `uri`, and similar formats. Numeric and array bounds (`minimum`, `maximum`, `minItems`, `maxItems`) are honored, and `$ref` references are resolved before generation.

OpenAPI-generated endpoints carry no conditions. When the same `(method, path)` is defined in both a JSON endpoint file and an OpenAPI spec, **the JSON entry wins** and a warning is logged at startup. This lets you import a whole spec for shape and then hand-tune individual endpoints with custom responses, conditions, or status codes.

The OpenAPI folder is optional â€” if it is missing or empty, Boggrt simply registers no OpenAPI-derived routes.

## Common use cases

- Stub third-party APIs during local development
- Spin up a mock server from an existing OpenAPI spec before the real backend is ready
- Run deterministic integration tests with Testcontainers
- Share the same mock definitions between CI jobs and Docker Compose environments
- Validate different request payloads with condition-based responses
